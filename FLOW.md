采购商品推荐与 Word 自动填充系统流程文档
1. 本文档作用
本文档用于规定系统的数据流和主流程。
系统开发过程中，所有功能都必须服务这条主线：
上传 Word → 解析采购参数 → 获取商品数据 → 匹配排序 → 生成响应内容 → 写回 Word → 下载结果文档。
如果某个功能不能明确接入这条流程，第一版暂时不做。
2. 总体数据流
第一版系统数据流如下：
input.docx
↓
requirements.json
↓
products.json
↓
clean_products.json
↓
ranked_products.json
↓
responses.json
↓
output.docx
每一步都要能单独检查。
这样做的目的：
1.	出问题时能快速定位是哪一步坏了。
2.	Codex 每次只改一个环节，避免全局乱改。
3.	后续淘宝、京东搜索模块可以替换商品数据来源，不影响后面的评分和写回流程。
3. 流程 1：上传 Word
输入
用户上传的采购技术规范 Word 文档。
第一版优先适配：
办公屏风01模板样例.docx
处理内容
系统接收 .docx 文件，并保存为临时输入文件。
输出
input.docx
验收标准
1.	页面可以上传 .docx 文件。
2.	上传后系统能识别文件名。
3.	文件能被后续文档解析模块读取。
4. 流程 2：解析技术参数表
输入
input.docx
处理内容
系统读取 Word 中的技术参数特性表，提取每一行采购参数。
需要读取的字段：
1.	序号
2.	技术参数名称
3.	单位
4.	项目需求值或表述
5.	投标人响应值
输出
requirements.json
requirements.json 示例结构
[
  {
    "index": "1",
    "name": "尺寸要求（长×宽×高）",
    "unit": "mm",
    "required_value": "可按项目调整，1600×750×750",
    "response_value": ""
  },
  {
    "index": "2",
    "name": "主体材质",
    "unit": "-",
    "required_value": "钢制框架 + 实木颗粒板面板",
    "response_value": ""
  }
]
验收标准
1.	系统能读出技术参数表。
2.	页面能展示解析结果。
3.	每一行参数的名称和需求值不能丢失。
4.	解析结果可以保存为 requirements.json。
5. 流程 3：获取候选商品数据
输入
requirements.json
处理内容
系统根据采购参数获取候选商品。
第一版优先使用本地缓存商品数据：
products.json
后续可以增加：
1.	京东搜索。
2.	淘宝搜索。
3.	手动导入商品链接。
4.	手动导入商品文本。
输出
products.json
products.json 示例结构
[
  {
    "platform": "京东",
    "title": "办公屏风工位隔断桌 1600×750×750",
    "price": 899,
    "shop": "示例店铺",
    "url": "https://example.com/product/1",
    "image_url": "",
    "specs_text": "尺寸：1600×750×750mm；材质：钢制框架，板式桌面；支持配送安装",
    "service_text": "配送至指定地点，支持安装",
    "raw_text": "完整商品原始文本"
  }
]
验收标准
1.	系统能读取 products.json。
2.	页面能展示商品列表。
3.	每个商品至少包含 title、price、platform、url、specs_text。
4.	至少准备 5 到 10 条办公屏风相关商品数据。
5.	可以混入 1 到 2 条无关商品，用于测试排序是否有效。
6. 流程 4：清洗商品数据
输入
products.json
处理内容
将不同来源的商品信息整理成统一格式。
统一字段：
1.	platform
2.	title
3.	price
4.	shop
5.	url
6.	image_url
7.	specs_text
8.	service_text
9.	raw_text
输出
clean_products.json
验收标准
1.	商品字段统一。
2.	缺失字段用空字符串或 null 表示。
3.	价格字段尽量转成数字。
4.	商品链接保留。
5.	商品参数文本保留。
7. 流程 5：匹配评分排序
输入
requirements.json
clean_products.json
处理内容
系统根据采购参数对候选商品进行打分。
第一版使用规则评分。
建议评分维度：
标题相关度：25 分
尺寸匹配度：20 分
材质匹配度：15 分
服务匹配度：10 分
参数完整度：10 分
价格合理性：10 分
风险项数量：10 分
示例判断：
1.	标题包含“办公屏风”，加分。
2.	标题包含“工位”或“隔断”，加分。
3.	参数中出现 1600、750、750，加分。
4.	参数中出现钢制、板材、颗粒板、玻璃等关键词，加分。
5.	服务文本中出现配送、安装，加分。
6.	参数缺失较多，扣分。
7.	商品明显无关，降权。
输出
ranked_products.json
ranked_products.json 示例结构
[
  {
    "platform": "京东",
    "title": "办公屏风工位隔断桌 1600×750×750",
    "price": 899,
    "url": "https://example.com/product/1",
    "score": 86,
    "reasons": [
      "标题包含办公屏风和工位",
      "参数中出现 1600×750×750",
      "服务文本包含配送安装"
    ],
    "risks": [
      "环保参数未明确标注，需人工复核"
    ]
  }
]
验收标准
1.	每个商品都有 score。
2.	每个商品都有 reasons。
3.	有风险的商品显示 risks。
4.	明显相关商品排在前面。
5.	明显无关商品排在后面。
8. 流程 6：生成投标人响应内容
输入
requirements.json
ranked_products.json
处理内容
系统选择 Top 商品，并根据采购参数生成“投标人响应值”。
生成原则：
1.	能从商品信息确认的参数，写明确响应。
2.	不能确认的参数，写“需人工复核”。
3.	不能编造商品页面没有的信息。
4.	响应文字要适合写入 Word 表格。
输出
responses.json
responses.json 示例结构
[
  {
    "index": "1",
    "name": "尺寸要求（长×宽×高）",
    "response_value": "候选商品页面显示尺寸为 1600×750×750mm，与项目需求值基本一致。"
  },
  {
    "index": "13",
    "name": "环保要求",
    "response_value": "商品页面未明确标注甲醛释放量参数，建议人工复核。"
  }
]
验收标准
1.	每一条采购参数都有响应内容。
2.	响应内容不能为空。
3.	无法确认的字段必须标明“需人工复核”。
4.	响应内容能写进 Word 表格。
9. 流程 7：写回 Word
输入
input.docx
responses.json
ranked_products.json
处理内容
系统将 responses.json 写回原 Word 的“投标人响应值”列。
同时在文档末尾追加“推荐商品汇总表”。
推荐商品汇总表包含：
1.	排名
2.	商品名称
3.	平台
4.	价格
5.	链接
6.	匹配分数
7.	匹配理由
8.	风险提示
输出
output.docx
验收标准
1.	输出 Word 可以正常打开。
2.	原始文档结构尽量保持。
3.	技术参数表中的“投标人响应值”列被填充。
4.	文档末尾出现推荐商品汇总表。
5.	商品链接、分数、理由和风险提示能看到。
10. 流程 8：下载结果文档
输入
output.docx
处理内容
页面提供下载按钮，用户可以下载生成后的 Word 文档。
输出
用户下载到本地的新 Word 文档。
验收标准
1.	页面能下载 output.docx。
2.	下载后的文档可以打开。
3.	内容包含填充后的响应值和推荐商品汇总表。
11. 第一版核心闭环
第一版必须跑通：
上传 Word
↓
解析技术参数表
↓
读取 products.json
↓
商品打分排序
↓
生成响应内容
↓
写回 Word
↓
下载 output.docx
12. 搜索增强流程
淘宝、京东搜索属于后续增强。
增强流程：
采购参数
↓
生成搜索关键词
↓
打开京东或淘宝搜索页
↓
提取商品标题、价格、店铺、链接、图片
↓
转成 products.json 格式
↓
进入后续清洗、评分、写回流程
搜索失败时，系统必须能回到缓存 JSON 流程。
13. 当前开发优先级
当前只做阶段 1：
Word 输入输出闭环
在阶段 1 完成前，不做淘宝搜索，不做京东搜索，不接 AI，不做复杂图片识别。
