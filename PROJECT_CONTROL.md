采购商品推荐与 Word 自动填充系统项目总控文档

## 1. 项目定位

本项目定位为客户交付版采购商品推荐与 Word 自动填充系统。

系统目标是从采购技术规范 Word 中解析需求，获取候选商品数据，标准化商品字段，进行规则评分排序，生成 Top 候选商品与参数响应，并写回 Word，形成可交付的采购辅助文档。

本系统服务于采购辅助、资料整理、候选商品筛选和文档填充，不直接替代人工采购判断。所有推荐结果、采集结果、参数响应和风险提示均需人工复核，不能称为最终采购结论。

项目已经完成阶段 1 到阶段 7，具备 Word 读取、Word 写入、需求解析、本地商品读取与标准化、规则评分排序、Top 3 推荐、规则响应生成、Word 推荐结果写回，以及 `main.py` 端到端流程。下一阶段进入生产级商品数据采集子系统建设。

## 2. 当前验收对象

当前验收对象为客户或客户指定验收人员。

验收重点包括：

1. 端到端流程可运行。
2. 商品数据采集可配置。
3. 采集结果可追踪。
4. 失败有日志和报告。
5. Word 输出可打开。
6. 推荐依据和风险提示清楚。
7. 无法确认的参数标记“需人工复核”。
8. 不把规则推荐结果称为最终采购结论。

## 3. 适配范围与交付边界

当前系统优先适配采购技术规范 Word 文档中的技术参数表，尤其是已经验证过的办公屏风类样例模板。后续可扩展到更多采购品类、更多 Word 模板和更多客户指定数据源。

客户交付版强调可配置、可追踪、可验收的工程闭环，不承诺对任意平台、任意页面、任意登录环境进行万能采集。对于复杂平台、登录页面、验证码页面、风控页面和授权不明确的数据源，系统必须保留安全边界，并提供本地 JSON 或手动导入作为兜底。

## 4. 系统输入

系统输入包括：

1. 采购技术规范 Word。
2. 本地测试商品 JSON。
3. 手动导入商品数据。
4. 客户提供的公开 URL 列表。
5. 客户授权数据源。
6. 后续可扩展的平台适配器。

输入数据必须明确来源。采集类输入需要记录数据源配置、seed URL、请求结果、解析结果和失败原因，方便客户验收和问题追踪。

## 5. 系统输出

系统输出包括：

1. `recommendation_result.docx`
2. `requirements.json`
3. `crawler_raw.json`
4. `crawler_products.json`
5. `crawler_report.json`
6. `ranked_products.json`
7. `responses.json`
8. `final_report.json` 或 `final_report.md`，后续阶段可做
9. `crawler_run.log`

`recommendation_result.docx` 是客户可查看的采购辅助输出文档。JSON 和日志文件用于阶段验收、问题定位、证据追踪和人工复核。

## 6. 客户交付主流程

客户交付主流程如下：

```text
Word 输入
→ 需求解析
→ 数据源选择
→ 商品采集或导入
→ 商品标准化
→ 去重与清洗
→ 规则评分排序
→ Top 3 候选
→ 参数响应生成
→ Word 写回
→ 采集报告与风险报告
→ 客户验收输出
```

每一步都必须有清晰的输入、输出、失败兜底和验收方式。任何无法确认的商品参数，必须在响应内容、风险提示或报告中标记“需人工复核”。

## 7. 数据流

客户交付版数据流：

```text
input.docx
→ requirements.json
→ seed_urls.json 或 sample_products.json
→ crawler_raw.json
→ crawler_products.json
→ crawler_report.json
→ ranked_products.json
→ responses.json
→ recommendation_result.docx
→ final_report.json 或 final_report.md
→ crawler_run.log
```

说明：

1. `requirements.json` 保存从 Word 解析出的采购需求。
2. `sample_products.json` 是本地测试或兜底商品数据。
3. `seed_urls.json` 保存客户提供或配置生成的公开 URL。
4. `crawler_raw.json` 保存原始采集结果或页面抽取结果。
5. `crawler_products.json` 保存采集并标准化后的商品数据。
6. `crawler_report.json` 保存成功、失败、跳过、robots 限制、HTTP 错误、解析失败等信息。
7. `ranked_products.json` 保存规则评分排序结果。
8. `responses.json` 保存写回 Word 的参数响应。
9. `crawler_run.log` 保存采集运行日志。

## 8. 模块拆分

### 8.1 Word 读取模块

职责：读取采购技术规范 Word，定位技术参数表，提取采购需求。

输入：`.docx` 文件。

输出：结构化采购需求列表，保存为 `requirements.json`。

### 8.2 Word 写回模块

职责：将 `responses.json` 写回 Word 的投标人响应值列，并追加推荐商品汇总、推荐依据和风险提示。

输入：原始 Word、`responses.json`、`ranked_products.json`。

输出：`recommendation_result.docx`。

### 8.3 需求解析模块

职责：将 Word 表格中的参数名称、单位、需求值、原响应值转换为统一结构，供后续商品匹配和响应生成使用。

输出字段至少包括：

1. `index`
2. `name`
3. `unit`
4. `required_value`
5. `response_value`

### 8.4 商品数据导入模块

职责：读取本地测试商品 JSON、手动导入商品数据和客户授权数据源产物，形成统一候选商品输入。

该模块必须保留本地 JSON 和手动导入作为兜底，不能把系统可用性完全绑定到外部平台实时采集。

### 8.5 生产级采集配置模块

职责：管理采集开关、数据源类型、客户授权说明、请求限速、超时、重试、输出路径、字段映射和验收配置。

输出：可审查、可复现的采集配置。

### 8.6 seed URL 模块

职责：读取客户提供的公开 URL 列表或配置生成的初始 URL，校验格式，去重，并输出 `seed_urls.json`。

该模块不得绕过登录、验证码或平台访问限制。

### 8.7 robots 检查模块

职责：在抓取前检查目标站点 robots 规则和允许范围，记录允许、拒绝、跳过原因。

输出：写入 `crawler_report.json` 和 `crawler_run.log`。

### 8.8 fetcher 模块

职责：按配置限速请求公开页面或客户授权数据源，记录 HTTP 状态、错误原因、重试次数和原始响应摘要。

要求：可配置、可限速、可追踪、可验收。

### 8.9 parser 模块

职责：从原始页面或数据源响应中提取商品标题、价格、店铺、链接、图片、参数文本、服务文本和原始文本。

解析失败必须记录原因，不能静默丢弃。

### 8.10 normalizer 模块

职责：将不同来源的商品数据统一为标准字段。

统一字段包括：

1. `platform`
2. `title`
3. `price`
4. `shop`
5. `url`
6. `image_url`
7. `specs_text`
8. `service_text`
9. `raw_text`
10. `source`
11. `evidence`
12. `manual_review_required`

### 8.11 deduper 模块

职责：按 URL、标题、店铺、规格摘要等信息进行去重与清洗，保留证据来源和被合并记录。

输出：去重后的 `crawler_products.json`。

### 8.12 pipeline 模块

职责：串联配置、seed URL、robots 检查、fetcher、parser、normalizer、deduper、报告和日志，形成可验收的采集流水线。

### 8.13 logging/report 模块

职责：生成 `crawler_report.json` 和 `crawler_run.log`，记录采集数量、成功数量、失败数量、跳过数量、robots 限制、HTTP 错误、解析失败、字段缺失、人工复核项和风险提示。

### 8.14 规则评分排序模块

职责：根据采购需求和标准化商品字段进行规则评分排序，输出 `ranked_products.json`。

评分结果必须附带理由和风险提示，不能把排序结果描述为最终采购结论。

### 8.15 Top 3 推荐与响应生成模块

职责：选择 Top 3 候选商品，并根据采购需求生成 Word 响应内容。

原则：

1. 只能根据已知商品字段和证据生成。
2. 不能编造商品页面或数据源没有的信息。
3. 无法确认的参数必须写“需人工复核”。

### 8.16 main.py 端到端流程

职责：串联已完成模块，支持从输入 Word 到推荐结果 Word 的端到端运行。

当前 `main.py` 端到端流程已完成并验收，阶段 8 不应删除旧接口或破坏既有闭环。

## 11. 风险与兜底方案

风险 1：外部网站出现登录、验证码、风控或 robots 限制。

兜底方案：记录失败原因，跳过受限页面，使用本地 JSON 或手动导入数据继续后续推荐流程。

风险 2：商品详情页参数不完整。

兜底方案：缺失字段进入风险提示，输出“需人工复核”，不能补写没有证据的参数。

风险 3：页面结构变化导致解析失败。

兜底方案：记录解析失败、保留原始摘要、输出失败报告，并允许切换 parser 或手动导入。

风险 4：客户数据源授权范围不清。

兜底方案：不采集授权不明确的数据源，要求客户提供公开 URL 列表、授权说明或离线数据。

风险 5：规则推荐被误解为采购结论。

兜底方案：所有文档、报告和输出中都必须说明结果为采购辅助候选，需人工复核，不是最终采购结论。

## 12. 严格红线

以下事项严格禁止：

1. 不绕登录。
2. 不绕验证码。
3. 不绕平台风控。
4. 不偷用 Cookie。
5. 不使用代理池规避封禁。
6. 不抓取个人信息。
7. 不自动下单。
8. 不承诺万能平台采集。
9. 不把采集结果或推荐结果称为最终采购结论。
